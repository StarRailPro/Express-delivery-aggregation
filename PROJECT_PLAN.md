> 版本：v1.0 | 制定日期：2026-04-15 | 状态：待评审

## 一、项目背景与目标

### 1.1 项目背景

随着电商行业的蓬勃发展，中国快递业务量持续增长，2025 年全国快递业务量已突破 1800 亿件。消费者日常在淘宝、京东、拼多多、抖音等多个平台购物后，快递分散在不同物流体系中，用户需要：

- 辗转多个 App（菜鸟、京东物流、极兔等）分别查看物流
- 无法在同一界面对比所有快递的运输进度
- 缺乏直观的地理位置感知，难以判断快递距离自己的远近

**核心痛点**：物流信息碎片化，缺乏统一聚合与可视化展示方案。

### 1.2 项目目标

#### 1.2.1 核心目标

构建一个快递信息聚合平台，用户只需将快递单号上传至本平台，即可：

1. **统一查询**：在一个界面中查看所有快递的实时物流信息
2. **地图可视化**：通过中国地图直观展示快递当前位置与运输轨迹
3. **智能管理**：自动识别快递公司、定时刷新、签收归档

#### 1.2.2 阶段目标

| 阶段              | 目标                             | 核心交付物    | 预计周期 |
| --------------- | ------------------------------ | -------- | ---- |
| Phase 1 - MVP   | 搭建前后端骨架，接入快递查询 API，实现单号查询与基础展示 | 可运行的查询系统 | 3 周  |
| Phase 2 - 地图可视化 | 集成高德地图，实现城市级定位与运输轨迹展示          | 地图可视化功能  | 2 周  |
| Phase 3 - 体验优化  | 定时刷新、状态通知、快递归档、移动端适配           | 完整可用产品   | 2 周  |

#### 1.2.3 成功指标

| 指标        | 目标值       | 衡量方式            |
| --------- | --------- | --------------- |
| 快递单号识别准确率 | ≥ 95%     | 自动识别 vs 手动选择对比  |
| 物流信息查询成功率 | ≥ 98%     | API 调用成功 / 总调用  |
| 页面首屏加载时间  | ≤ 2s      | Lighthouse 性能评分 |
| 地图标注准确率   | ≥ 90%     | 城市解析结果人工抽检      |
| 用户满意度     | ≥ 4.0/5.0 | 内测用户反馈          |

## 二、核心功能模块划分

### 2.1 模块架构总览

```
┌─────────────────────────────────────────────────────────────┐
│                      快递信息聚合平台                         │
├──────────┬──────────┬──────────┬──────────┬────────────────┤
│  用户模块  │  快递模块  │  地图模块  │  刷新模块  │  系统管理模块  │
├──────────┴──────────┴──────────┴──────────┴────────────────┤
│                    公共服务层                                  │
│     认证鉴权 │ 错误处理 │ 日志记录 │ API 调用管理 │ 缓存服务     │
├─────────────────────────────────────────────────────────────┤
│                    数据持久层                                  │
│              MongoDB (users | packages | tracking_records)    │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 模块详细说明

#### 模块 A：用户模块

| 子功能         | 描述                                        | 优先级 | 前置依赖 |
| ----------- | ----------------------------------------- | --- | ---- |
| A1 用户注册     | 用户名 + 密码注册，密码 bcrypt 加密（salt rounds ≥ 10） | P0  | 无    |
| A2 用户登录     | JWT Token 认证，Token 有效期 7 天                | P0  | A1   |
| A3 用户信息     | 获取当前用户信息，含快递统计                            | P0  | A2   |
| A4 Token 刷新 | Token 过期前自动续期                             | P1  | A2   |

#### 模块 B：快递模块

| 子功能     | 描述                        | 优先级 | 前置依赖   |
| ------- | ------------------------- | --- | ------ |
| B1 添加快递 | 输入单号，自动识别快递公司并创建记录        | P0  | A2, D1 |
| B2 快递列表 | 按状态分组展示用户所有快递（运输中/已签收/异常） | P0  | A2     |
| B3 物流详情 | 展示选中快递的完整物流轨迹时间线          | P0  | B2     |
| B4 快递别名 | 用户为快递设置自定义名称              | P1  | B1     |
| B5 手动刷新 | 用户主动触发物流信息更新              | P0  | B1, D2 |
| B6 编辑快递 | 修改别名等非核心字段                | P1  | B1     |
| B7 删除快递 | 从列表中移除快递记录                | P0  | B1     |
| B8 快递归档 | 签收快递移出主列表，归档管理            | P2  | B2     |

#### 模块 C：地图模块

| 子功能      | 描述                       | 优先级 | 前置依赖   |
| -------- | ------------------------ | --- | ------ |
| C1 地图初始化 | 集成高德地图 JS API 2.0，渲染中国地图 | P0  | 无      |
| C2 城市标注  | 在地图上标注快递当前所在城市           | P0  | C1, E2 |
| C3 运输轨迹  | 绘制快递从出发地到当前位置的路线         | P1  | C2     |
| C4 多快递展示 | 地图上同时显示所有快递的当前位置         | P1  | C2     |
| C5 轨迹回放  | 动画回放快递的完整运输路径            | P2  | C3     |

#### 模块 D：刷新模块

| 子功能       | 描述                       | 优先级 | 前置依赖 |
| --------- | ------------------------ | --- | ---- |
| D1 快递公司识别 | 调用快递鸟/快递100 API 自动识别快递公司 | P0  | 无    |
| D2 物流轨迹查询 | 调用快递 API 获取完整物流轨迹        | P0  | D1   |
| D3 定时刷新   | 后台每 2 小时自动刷新未签收快递        | P1  | D2   |
| D4 签收降频   | 签收后自动停止刷新                | P1  | D3   |
| D5 手动刷新   | 用户触发即时刷新，不受定时限制          | P0  | D2   |

#### 模块 E：地址解析模块

| 子功能          | 描述                             | 优先级 | 前置依赖 |
| ------------ | ------------------------------ | --- | ---- |
| E1 文本解析      | 从物流描述中正则提取城市名（匹配 `【xxx市】` 等模式） | P0  | 无    |
| E2 Geocoding | 调用高德 Geocoding API 将城市名转为经纬度   | P0  | E1   |
| E3 坐标缓存      | 城市名 → 坐标映射缓存，避免重复查询            | P1  | E2   |

#### 模块 F：系统管理模块

| 子功能         | 描述                      | 优先级 | 前置依赖       |
| ----------- | ----------------------- | --- | ---------- |
| F1 API 调用统计 | 记录每日第三方 API 调用量         | P1  | D1, D2, E2 |
| F2 错误监控     | 统一错误日志记录与告警             | P1  | 无          |
| F3 识别缓存     | 快递公司识别结果缓存（相同前缀单号不重复识别） | P1  | D1         |

### 7.2 接口质量

| 措施     | 具体要求                                          |
| ------ | --------------------------------------------- |

| 错误码规范  | 2xx 成功、4xx 客户端错误、5xx 服务端错误                    |

### 7.3 安全保障

| 措施         | 具体要求                                                     |

| 输入防注入      | Mongoose Schema 校验 + 手动参数校验                              |
| CORS 配置    | 生产环境限制允许的源                                               |

### 9.8 代码规范

- 全栈统一 TypeScript，禁止使用 any（除非明确标注原因）
- ESLint + Prettier 统一代码风格
- 组件命名：PascalCase（如 `PackageList`）
- 函数/变量命名：camelCase（如 `fetchTrackingInfo`）
- 常量命名：UPPER\_SNAKE\_CASE（如 `MAX_RETRY_COUNT`）
- 文件命名：组件用 PascalCase，工具/服务用 camelCase
- Git Commit 格式：`<type>: <description>`（feat / fix / docs / refactor / chore）

### 9.9 安全规范

- 所有 API Key 必须存放在 .env 文件中，禁止提交到版本库
- .env 文件必须加入 .gitignore
- 用户密码使用 bcrypt 加密存储，salt rounds ≥ 10
- JWT Token 设置合理过期时间（7 天）
- API 接口需做参数校验，防止注入攻击

### 9.10 API 调用管理

- 后端统一管理第三方 API 调用，前端不直接调用快递/高德服务端 API
- 实现 API 调用计数器，记录每日调用量
- 快递公司识别结果做缓存，相同前缀的单号不重复调用识别接口
- Geocoding 结果做缓存，相同城市名不重复查询

### 9.11 错误处理

- 后端统一错误响应格式：`{ code: number, message: string, data?: any }`
- 第三方 API 调用失败时做重试（最多 3 次，指数退避）
- 前端全局错误提示，避免白屏
